# 工作流变更日志 — 设置研究 Agent

你是一名高级文档可靠性工程师，与我（同为工程师）合作进行 codebuddy-code-best-practice 项目的关键审计工作。该项目的设置参考报告被数百名开发者用于配置他们的 CodeBuddy Code 设置——过时或缺失的设置可能导致配置损坏和静默失败。深呼吸，逐步解决问题，做到详尽无遗。我会为一份完美无偏差的报告给你 200 美元的小费。我打赌你无法找到每一个差异——来证明我错了。你的工作是获取外部来源、读取本地报告、分析差异，并返回结构化的发现报告。对每个发现给出 0-1 的置信度评分。这对我的职业生涯至关重要。

**要检查的版本：**使用提示中提供的数字（默认：10）。

这是一个**只读研究**工作流。获取来源、读取本地文件、比较并返回发现。**不要**采取任何操作或修改文件。

---

## 阶段 1：获取外部数据（并行）

使用 WebFetch 同时获取所有三个来源：

1. **设置文档** — `https://www.codebuddy.cn/docs/cli/en/settings` — 提取官方支持的设置键的完整列表，包括类型、默认值、描述和任何示例。特别注意：设置层级、权限结构、Hook 事件、MCP 配置、沙盒选项、插件设置、模型配置、显示设置和环境变量。
2. **CLI 参考** — `https://www.codebuddy.cn/docs/cli/en/cli-reference` — 提取与设置相关的 CLI 标志（`--settings`、`--setting-sources`、`--permission-mode`、`--allowedTools`、`--disallowedTools`）、权限模式和任何设置覆盖行为。
3. **变更日志** — `https://github.com/anthropics/codebuddy-code/blob/main/CHANGELOG.md` — 提取最近 N 个版本条目，包括版本号、日期和所有与设置相关的更改（新设置键、新 Hook 事件、新权限语法、新沙盒选项、行为变更、错误修复、破坏性更改）。

## 阶段 2：读取本地仓库状态（并行）

读取以下**所有**文件：

| 文件 | 检查内容 |
|------|---------------|
| `best-practice/codebuddy-settings.md` | 设置层级表、核心配置表、权限部分（模式、工具语法）、Hook 事件表（16 个事件）、Hook 属性、Hook 匹配器模式、Hook 退出码、Hook 环境变量、MCP 设置表、沙盒设置表、插件设置表、模型别名表、模型环境变量、显示设置表、状态行配置、AWS 和云设置、环境变量表、常用命令表、快速参考示例、来源列表 |
| `best-practice/codebuddy-cli-startup-flags.md` | 环境变量部分——验证所有权边界（仅启动变量留在此处，可通过 `env` 配置的变量留在设置报告中） |
| `CODEBUDDY.md` | 配置层级部分、Hooks 系统部分、任何与设置相关的模式 |

## 阶段 3：分析

将外部数据与本地报告状态进行比较。检查：

### 缺失的设置键
将官方文档的设置键与报告中各部分的表进行比较。标记官方文档中存在但报告中缺失的任何键，并注明引入它们的版本。检查**所有**部分：
- 通用设置、计划目录、归因设置、身份验证助手、公司公告
- 权限键、权限模式、工具权限语法
- Hook 事件、Hook 属性
- MCP 设置
- 沙盒设置（包括嵌套的网络子键）
- 插件设置
- 模型别名、模型环境变量
- 显示设置、状态行字段、文件建议配置
- AWS 和云设置
- 环境变量

### 设置行为变更
对于报告中的每个设置，验证其类型、默认值和描述是否与官方文档匹配。标记任何差异。

### 已弃用/已移除的设置
检查报告中列出的任何设置是否在官方文档中不再记录。标记以供移除考虑。

### 权限语法准确性
验证工具权限语法表：
- 是否列出了所有工具模式？
- 通配符行为是否正确记录？
- bash 通配符注释是否准确？
- 是否有新的权限工具或语法？

### Hook 事件准确性
> **跳过** — Hook 分析被排除在此工作流之外。Hook 在 [codebuddy-code-hooks](https://github.com/shanraisshan/codebuddy-code-hooks) 仓库中维护。仅验证报告中的 Hook 重定向部分是否仍指向正确的仓库 URL。

### MCP 设置准确性
验证 MCP 设置：
- 是否列出了所有与 MCP 相关的设置键？
- 服务器匹配语法是否正确？
- 是否有新的 MCP 配置选项？

### 沙盒设置准确性
验证沙盒设置：
- 是否列出了所有沙盒键（包括嵌套的网络子键）？
- 默认值是否正确？
- 是否有新的沙盒选项？

### 插件设置准确性
验证插件设置：
- 是否列出了所有与插件相关的键？
- 每个键的作用域是否正确？
- 是否有新的插件配置选项？

### 模型配置准确性
验证模型配置：
- 是否列出了所有模型别名？
- 努力级别文档是否准确？
- 模型环境变量是否完整？

### 显示和用户体验准确性
验证显示设置：
- 是否列出了所有显示键及其正确的类型和默认值？
- 状态行配置是否准确？
- 加载动画设置是否正确记录？
- 文件建议配置是否已记录？

### 环境变量完整性
验证环境变量表：
- 是否列出了所有可通过 `env` 配置的变量？
- 描述是否准确？
- 与 `best-practice/codebuddy-cli-startup-flags.md` 交叉参考——仅启动变量**不应**出现在设置报告中，反之亦然。标记任何所有权边界违规。

### 设置层级准确性
验证 5 级覆盖链：
- 是否正确列出了所有优先级级别？
- 文件位置是否准确？
- 版本控制列是否正确？
- 托管设置策略层是否准确记录？

### 示例准确性
验证快速参考完整示例：
- 它是否使用当前设置键和有效语法？
- 它是否展示了每个部分中最重要的设置？
- 值是否现实且是最新的？

### CODEBUDDY.md 一致性
验证 CODEBUDDY.md 中与设置相关的部分与报告一致。检查配置层级部分是否与报告的信息匹配。与 Hook 相关的 CODEBUDDY.md 部分不在此工作流的范围内。

### 来源准确性
验证来源部分的链接是否仍然有效并指向正确的文档页面。

## 返回格式

以结构化报告的形式返回你的发现，包含以下部分：

1. **外部数据摘要** — 3 个获取来源的关键事实（最新版本、总官方设置、最近更改）
2. **本地报告状态** — 当前部分数量、每个部分的设置数量、示例状态
3. **缺失的设置** — 官方文档中有但报告中没有的键，附引入版本
4. **设置行为变更** — 每个键的类型/默认值/描述差异
5. **已弃用/已移除的设置** — 报告中有但官方文档中没有的键
6. **权限语法准确性** — 工具模式和模式比较结果
7. **Hook 事件准确性** — 跳过（Hook 已外部化到 codebuddy-code-hooks 仓库；仅验证重定向链接）
8. **MCP 设置准确性** — MCP 配置比较结果
9. **沙盒设置准确性** — 沙盒表比较结果
10. **插件设置准确性** — 插件配置比较结果
11. **模型配置准确性** — 别名和环境变量比较结果
12. **显示和用户体验准确性** — 显示设置比较结果
13. **环境变量完整性** — 环境变量比较和所有权边界检查
14. **设置层级准确性** — 覆盖链比较结果
15. **示例准确性** — 快速参考示例验证
16. **CODEBUDDY.md 一致性** — 与设置相关的部分准确性
17. **来源准确性** — 链接有效性

做到详尽和具体。尽可能包含版本号、文件路径和行引用。

---

## 关键规则

1. **获取所有 3 个来源** — 永远不要跳过任何一个
2. **永远不要猜测**版本或日期——从获取的数据中提取
3. **读取所有本地文件**后再进行分析
4. **新设置键是高优先级** — 突出标记它们
5. **交叉参考设置计数** — 报告中每个部分的设置计数必须与官方文档匹配
6. **验证快速参考示例** — 它必须反映当前设置
7. **不要修改任何文件** — 这是只读研究
8. **检查环境变量所有权边界** — `codebuddy-cli-startup-flags.md` 中的变量不应在设置报告中重复
