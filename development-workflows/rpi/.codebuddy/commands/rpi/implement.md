description: 执行带验证门控的分阶段实现
argument-hint: "<feature-slug> [--phase N] [--validate-only]"
---

## 用户输入

```text
$ARGUMENTS
```

你**必须**解析用户输入以提取特性标识（`rpi/` 中的文件夹名称）。

## 用途

此命令基于规划文档执行特性的分阶段实现。它协调专业代理，强制执行验证门控，并在整个实现过程中确保合规性。

**前提条件**：
- 特性文件夹存在于 `rpi/{feature-slug}/`
- 规划已完成（`rpi/{feature-slug}/plan/PLAN.md` 存在）

**输出位置**：`rpi/{feature-slug}/implement/`

**这是 RPI 工作流的第 4 步**（最后一步 - 实际实现）。

## 标志

- `--phase N`：执行特定阶段编号（1-8），如果省略则从阶段 1 开始
- `--validate-only`：仅验证当前阶段，不进行实现
- `--skip-validation`：跳过验证门控并继续（请谨慎使用）

## 可用代理

所有代理使用 **Opus 模型** 以获得最高质量。

### 实现代理

| 代理 | 类型 | 使用场景 |
|-------|------|----------|
| `senior-software-engineer` | 自定义 | 所有实现任务 |

### 支持代理

| 代理 | 类型 | 用途 |
|-------|------|------|
| `Explore` | 内置 | 实现前的代码探索 |
| `code-reviewer` | 自定义 | 代码审查和质量验证 |
| `constitutional-validator` | 自定义 | 针对项目规范进行验证 |
| `documentation-analyst-writer` | 内置 | 文档生成 |

### 代理路由

所有实现任务由 `senior-software-engineer` 代理处理。

## 阶段 0：加载上下文和规则

**前提条件**：已从用户输入中解析特性标识

**流程**：

### 0.1 加载项目规范

1. 检查仓库中是否存在规范或原则文档
2. 如果存在，提取：
   - 技术约束（类型安全、测试、组件隔离）
   - 业务原则（质量标准、工作流）
   - 架构边界
3. 存储约束以在实现过程中强制执行

### 0.2 加载领域特定指南

根据要修改的文件，加载相关的项目指南：
- 检查组件特定的 README 文件
- 检查编码风格指南
- 检查测试要求文档

### 0.3 分析实现范围

1. 读取 `rpi/{feature-slug}/plan/PLAN.md`
2. 识别所有需要修改的文件
3. 将文件映射到实现代理

**输出**：
- 规范上下文摘要
- 已加载的领域规则
- 文件到代理的映射
- 阶段执行计划

**验证**：
- [ ] 已加载规范（如果存在）
- [ ] 已为受影响的文件加载领域规则
- [ ] 所有文件已映射到代理
- [ ] 已理解执行计划

## 分阶段实现工作流

### 阶段实现循环

对 PLAN.md 中的每个阶段：

```
┌─────────────────────────────────────────────────────────────────┐
│ 阶段 N: [阶段名称]                                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  1. 代码发现（探索代理）                                           │
│     └─→ 在修改前理解现有代码                                       │
│                                                                  │
│  2. 实现（senior-software-engineer）                              │
│     └─→ 实现阶段交付物                                            │
│                                                                  │
│  3. 自我验证                                                      │
│     └─→ 工程师根据阶段清单进行验证                                  │
│                                                                  │
│  4. 代码审查（code-reviewer 代理）                                 │
│     └─→ 安全性、正确性、可维护性                                   │
│                                                                  │
│  5. 用户验证门控                                                   │
│     └─→ 停止并请求用户批准                                        │
│         ├─→ 通过：进入下一阶段                                     │
│         ├─→ 有条件通过：记录问题，继续                              │
│         └─→ 失败：修复问题，重新验证                                │
│                                                                  │
│  6. 文档更新                                                      │
│     └─→ 更新 PLAN.md 中的阶段状态                                 │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 步骤 1：代码发现（每个阶段）

**代理**：Explore（内置，通过 Task 工具）

**目的**：在进行更改之前，将实现建立在代码现实的基础上。

**流程**：
1. 通过 Task 工具启动 Explore 代理，使用 `subagent_type="Explore"`
2. 请求分析受当前阶段影响的文件
3. 理解现有模式、集成点、约束

**探索代理提示**：
```
Analyze the codebase to prepare for implementing Phase N of [feature-name].

Files to be modified in this phase:
[List files from PLAN.md]

Investigate and document:

1. **Current Implementation**
   - How do these files currently work?
   - What patterns are used?
   - What functions/classes will be affected?

2. **集成点**
   - 哪些其他文件导入或使用了这些模块？
   - 哪些API或接口将会发生变化？
   - 哪些测试覆盖了这段代码？

3. **Dependencies**
   - What libraries are used?
   - What internal utilities are available?
   - What constraints exist from current code?

4. **应遵循的模式**
   - 这些文件中使用了什么编码风格？
   - 遵循了哪些命名规范？
   - 存在哪些错误处理模式？

5. **Risks and Considerations**
   - What could break if we change this?
   - What edge cases exist?
   - What backward compatibility concerns?

Provide a discovery summary to inform implementation.
```

**输出**：为实现代理提供的发现摘要

## 步骤 2：实现（每个阶段）

**代理**：senior-software-engineer

**流程**：
1. 使用 senior-software-engineer 代理
2. 提供步骤 1 的发现上下文
3. 实现阶段的所有交付物
4. 遵循规范约束和项目规则

**实现代理提示模板**：
```
Acting as the [agent-name] agent, implement Phase N deliverables for [feature-name].

## Context
- Constitutional Constraints: [from Phase 0]
- Domain Rules: [from Phase 0]
- Discovery Summary: [from Step 1]

## Phase N Deliverables
[List from PLAN.md]

## Files to Modify
[List files with specific changes from PLAN.md]

## 实现要求
1. 遵循发现阶段识别的现有代码模式
2. 遵守宪法约束（类型安全、测试等）
3. 遵循项目特定规则（如适用）
4. 为新功能编写测试
5. 包含适当的日志记录
6. 优雅地处理错误

## 质量检查清单
- [ ] 代码遵循现有模式
- [ ] 适用的地方添加了类型标注
- [ ] 测试已编写并通过
- [ ] 未对现有功能引入破坏性变更
- [ ] 为可观测性添加了日志记录
- [ ] 错误处理全面完善

Implement all deliverables and report what was done.
```

## 步骤 3：自我验证

**代理**：senior-software-engineer（与步骤 2 相同）

**流程**：
1. 代理根据阶段清单验证实现
2. 运行代码检查（使用项目配置的 linter）
3. 运行与更改相关的测试
4. 验证构建是否成功

**验证命令**（根据你的项目调整）：

```bash
# 运行 linter
[your-linter-command]

# 运行测试
[your-test-command]

# 构建/编译
[your-build-command]
```

**自我验证清单**：
- [ ] 所有交付物已实现
- [ ] 代码检查通过
- [ ] 测试通过
- [ ] 构建成功
- [ ] 现有测试无回归
- [ ] 遵守规范约束
- [ ] 遵循领域规则

## 步骤 4：代码审查

**代理**：code-reviewer（自定义，自动调用）

**流程**：
1. 调用 code-reviewer 代理审查更改
2. 关注正确性、安全性、可维护性
3. 在继续之前解决阻塞问题

**代码审查代理提示**：
```
Acting as the code-reviewer agent, review the Phase N implementation for [feature-name].

## Files Changed
[List modified files]

## Changes Made
[Summary of implementation]

## Review Focus
- Correctness & tests
- Security & dependency hygiene
- Architectural boundaries
- Clarity over cleverness

## Constitutional Constraints
[From Phase 0]

Provide review using standard output format.
```

**审查结论**：
- **已批准**：进入用户验证
- **已批准但有建议**：记录建议，继续
- **需要修改**：修复问题，重新审查

## 步骤 5：用户验证门控

**关键**：此步骤**需要**用户交互。**不要**自动继续。

**流程**：
1. 展示阶段交付物清单
2. 显示已实现的内容（更改的文件、添加的功能）
3. 展示 PLAN.md 中的验证标准
4. 显示代码审查结果
5. **停止并等待用户决定**

**验证请求格式**：
```
## 阶段 N 验证请求

### 已完成的交付物
- [x] [交付物 1] - [实现摘要]
- [x] [交付物 2] - [实现摘要]
- ...

### 更改的文件
| 文件 | 变更类型 | 行数 |
|------|----------|------|
| [file] | [添加/修改] | [±N] |

### 测试
- [x] 单元测试：通过
- [x] 集成测试：通过
- [x] 构建：成功

### 代码审查
- 结论：[已批准 / 已批准但有建议]
- 问题：[无 / 列表]

### 验证标准（来自 PLAN.md）
- [ ] [标准 1]
- [ ] [标准 2]
- ...

---

**请验证阶段 N：**
- **通过**：阶段完成，进入阶段 N+1
- **有条件通过**：记录以下问题，谨慎继续
- **失败**：指定需要修复的问题后再继续
```

**用户决定**：
- **通过**：进入下一阶段
- **有条件通过**：记录问题，进入下一阶段
- **失败**：修复问题，重新运行步骤 2-5

---

## 步骤 6：文档更新

**流程**：
1. 更新 `rpi/{feature-slug}/plan/PLAN.md` 中的阶段状态
2. 更新 `rpi/{feature-slug}/implement/IMPLEMENT.md` 中的验证结果
3. 将每个阶段的验证结果追加到 IMPLEMENT.md

### 阶段状态跟踪

更新 PLAN.md 中的复选框：
```markdown
- [ ] 阶段 N：未开始
- [~] 阶段 N：进行中
- [x] 阶段 N：已验证（通过）
- [!] 阶段 N：有条件通过（附注释）
- [-] 阶段 N：验证失败（需要返工）
```

### IMPLEMENT.md 模板

```markdown
# 实现记录

**特性**：[feature-slug]
**开始日期**：[日期]
**状态**：[进行中 / 已完成]

---

## 阶段 1：[阶段名称]

**日期**：[日期]
**结论**：[通过 / 有条件通过 / 失败]

### 交付物
- [x] [交付物 1]
- [x] [交付物 2]

### 更改的文件
[带行数统计的列表]

### 测试结果
[测试输出摘要]

### 代码审查
[审查结论和注释]

### 注意事项
[任何附加说明]
```

## 错误处理

### 实现失败

**如果实现失败**：
1. 记录具体的失败信息
2. 分析根本原因
3. 尝试替代方法（最多 2 次尝试）
4. 如果仍然失败，停止并请求用户指导
5. **不要**带着损坏的实现进入下一阶段

**消息**："实现失败：[错误]。已尝试 [N] 种方法。需要用户指导。"

### 测试失败

**如果测试失败**：
1. 分析失败原因（代码缺陷 vs 测试缺陷）
2. 修复问题
3. 重新运行测试
4. 如果持续失败，记录并询问用户
5. **不要**在测试失败的情况下标记阶段完成

**消息**："测试失败：[失败项]。已尝试修复但未成功。需要用户审查。"

### 构建失败

**如果构建失败**：
1. 检查类型错误
2. 检查缺失的导入
3. 检查语法错误
4. 修复并重新构建
5. 如果持续失败，上报给用户

**消息**："构建失败：[错误]。无法自动解决。"

### 代理失败

**如果代理失败或超时**：
1. 使用相同输入重试一次
2. 如果仍然失败，在没有该代理贡献的情况下继续
3. 在验证请求中记录缺失

**消息**："代理 [名称] 失败。在没有贡献的情况下继续。"

## 注意事项

### 何时使用此命令

- 在 `/rpi:plan` 生成 PLAN.md 之后
- 当需要带验证门控的分阶段实现时
- 对于需要结构化实现的特性

### 何时不使用此命令

- Bug 修复（过于重量级，直接修复即可）
- 非常简单的更改（<30 分钟的工作量）
- 探索性原型
- 仅文档更改

### 最佳实践

1. **先审查 PLAN.md**：了解你要实现什么
2. **信任代码发现**：让 Explore 代理为实现提供信息
3. **遵循现有模式**：让代码发现指导实现
4. **不要跳过验证**：门控的存在是为了尽早发现问题
5. **边做边记录**：每个阶段后更新状态
6. **遇到困难时提问**：询问比错误地继续更好

### RPI 工作流的一部分

4 步中的第 4 步（描述 → 研究 → 规划 → **实现**）

## 完成后操作

**重要**：完成实现（所有阶段或重大进展）后，**始终**提示用户压缩对话：

> **上下文管理**：此实现工作流消耗了大量上下文。为保存进度并释放空间，请运行：
>
> ```
> /compact
> ```
>
> 这将总结对话并保存实现状态，同时减少后续工作的 token 使用量。

**何时提示压缩**：
- 所有阶段完成后
- 完成每个主要阶段后（如果是多会话实现）
- 如果实现过程中上下文即将耗尽
