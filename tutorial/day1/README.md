# 第 1 天 — 你与 CodeBuddy Code 的第一次对话

[返回第 0 天（设置）](../day0/README.md)

---

你已经安装了 CodeBuddy Code。然后呢？本指南将带你了解使用它的三个层级 — 每一层级给你更多控制 CodeBuddy **如何**完成工作的能力。

可以把它想象成雇人：
1. **提示（Prompting）** = 向街上的陌生人问路
2. **Agents（代理）** = 雇一个总是以某种方式做事的专家
3. **Skills（技能）** = 这个专家拥有针对特定任务的专门训练

---

## 层级 1：提示（直接提问）

> 🧠 **可以把它想象成**给一个知识渊博的朋友发短信。你问"卡拉奇的天气怎么样？"他们会给你*一个*答案 — 但你不知道他们是查看了天气应用、看了窗外，还是只是凭记忆猜测的。

打开你的终端并输入 `codebuddy`。你现在就在对话中了。试试输入：

```
what is the weather in Karachi?
```

CodeBuddy 会回答 — 但**如何**回答是不可预测的。它可能会：
- 从训练数据中提取（可能已过时）
- 搜索网络（如果有网络工具可用）
- 给你一个通用答案而非实时数据

对于快速问题来说这完全没问题！但如果你需要**一致、可靠的结果**，仅仅提示是不够的。

### 提示效果很好的场景

- 关于你的代码库提问（"这个文件做什么？"）
- 编写或编辑文档（"重写这封邮件让它听起来更专业"）
- 头脑风暴（"给我 5 个这个活动的主题行"）
- 解释事情（"像我不是开发者一样解释这个错误消息"）

### 局限性

每次你问"天气怎么样？"时，CodeBuddy 可能用不同的方式获取数据 — 或者根本不获取真实数据。无法保证它两次使用相同的来源或方法。

---

## 层级 2：Agents（专家）

**Agent** 是 CodeBuddy 扮演特定角色 — 就像分配一个职位头衔。

> 🧠 **可以把它想象成**一个餐厅厨房。没有 Agent，你走进一个随机的厨房喊"给我做意大利面！" — 听到你的人可能煮方便面或做一顿五道菜的意大利餐。有了 Agent，你雇了一个**意大利面厨师**，他的工作描述写着：*"始终使用新鲜食材，始终煮到弹牙（al dente），始终以相同方式装盘。"* 现在你确切知道每次会得到什么。

同样的想法应用到 CodeBuddy 上：

> **没有 Agent：** 你问 CodeBuddy "迪拜的天气怎么样？"
> 它可能检查训练数据、搜索网络或做个最佳猜测。你不知道它会做什么。
>
> **有了 Agent：** 一个 `weather-agent` 有清晰的工作描述：
> *"始终从 Open-Meteo API 查询迪拜的天气。始终以特定格式返回温度。"*
> 同样的问题，同样的方法，每次都一样。

### 本仓库中的真实示例

这个仓库有一个 `weather-agent` — 它的全部工作就是获取迪拜的温度。以下是它与简单提示的区别：

| | 提示 | Agent |
|---|---|---|
| **来源** | 可能是任何地方 | 始终使用 Open-Meteo API |
| **位置** | CodeBuddy 随便选的 | 始终是迪拜（纬度: 25.2, 经度: 55.3） |
| **格式** | 随机段落 | 干净的温度 + 单位 |
| **一致性** | 每次不同 | 同样的方法，每次都一样 |

### 要点

Agents 给你**可预测性**。同样的问题 → 同样的方法 → 同样的质量。这就是优势 — 不是 Agents 更聪明，而是它们**始终如一**。

## 层级 3：Skills（技能）

**Skill** 是 Agent（或 CodeBuddy 本身）可以使用的特定能力。

> 🧠 **可以把它想象成**一个新员工的培训手册。当有人加入你的团队时，他们有一个角色（Agent），但他们也经过特定的培训模块 — 如何使用 CRM、如何写提案、如何主持站会。每个培训模块就是一个 **Skill**。角色告诉他们*是什么*；技能告诉他们*如何*做特定的事。

现在想想一个真实的人：

> **Shayan** 有很多技能：
> - 工程技能 — 会写代码
> - 游戏技能 — 懂游戏机制
> - 阅读技能 — 能消化和总结长文档
>
> 每个技能有自己的知识和方法。Shayan 为正确的任务使用正确的技能。

CodeBuddy 的工作方式相同。`weather-agent` 有一个叫做 `weather-fetcher` 的 Skill：

- **Agent**（`weather-agent`）= 拥有"天气播报员"职位的人
- **Skill**（`weather-fetcher`）= 关于*如何*获取天气数据的专门训练

Skill 包含精确的指令：
1. 调用这个特定的 API URL
2. 从响应中的这个特定字段提取温度
3. 以这个特定格式返回

### 为什么要分离 Agents 和 Skills？

因为**一个 Agent 可以有多个 Skills**，而且**一个 Skill 可以被多个 Agents 使用**。

例如，假设你创建：
- 一个 `daily-report-agent` 来总结你的一天
- 它可以使用 `weather-fetcher` Skill（获取天气）+ `calendar-reader` Skill（获取会议）+ `email-summarizer` Skill（获取邮件摘要）

Skills 是可复用的构建块。Agents 是使用它们的人。

## 综合运用

这是完整的图景：

```
层级 1: 提示（PROMPTING）
你 → "天气怎么样？" → CodeBuddy 某种方式搞定
                        （不可预测的方法）

层级 2: AGENTS
你 → 天气 Agent → 始终使用相同的方法
                   （可预测的方法）

层级 3: SKILLS
你 → 天气 Agent → 使用 weather-fetcher Skill
                   （可预测的方法 + 具体指令）
```

每个层级增加更多控制：

| 层级 | 你控制什么 | 最适合 |
|-------|---------|----------|
| **提示** | 问题 | 快速的一次性问题 |
| **Agents** | 问题 + 谁来回答 | 可重复的任务 |
| **Skills** | 问题 + 谁来回答 + 如何做 | 关键工作流 |

---

## 下一步？

目前，花时间在**层级 1** — 直接提问。习惯在终端中向 CodeBuddy 提问。你用得越多，就越会发现哪些任务适合使用 Agent。
